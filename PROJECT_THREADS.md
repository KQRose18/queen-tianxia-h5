## 打包与发布分叉

- Thread ID: `019eefaf-bb7d-7182-aa53-1427433303e1`
- 用途：发布包、静态托管、Cloudflare/Vercel/Netlify、外网体验链接
- 协作要求：每次打包发版前必须和主开发线程同步版本状态，确认是否允许发布、资源是否完整、哪些素材仍是占位、最新 release 包路径和外网链接。打包分叉不单独判断“可上线”，只执行主线程确认后的版本。
- 打包前检查：
  - 主线程是否明确说“可以打包/可以发布”。
  - `app.js`、`index.html`、`style.css` 是否为最新版本。
  - 所有 `assets/...png`、音频、manifest 引用是否存在。

- 打包后必须回传：
  - release 目录、zip 路径、包体大小、图片/音频压缩结果。

## 暂不新开的职能

- QA / 测试回归：先由主开发线程维护验收清单并执行。等外部试玩人数上来、版本改动变频繁后，再单独开“测试与反馈分叉”。
- 商业化 / 付费设计：当前由主开发线程兼任，只做月卡、剧情包、衣橱的 MVP 方案，不提前拆分。
- 数据分析 / 留存调优：当前没有真实埋点数据，先由主开发线程定义指标。上线后再决定是否单独开“数据与数值调优分叉”。

## 分叉协作规则

1. 主线程是唯一 PM 和版本闸口，负责玩法取舍、数值、接入、验收、是否发布。
2. 剧情和美术分叉交付“剧情 + 美术”成套资产，主线程负责接入和最终取舍。
3. 音效和 BGM 分叉交付“音频文件 + 场景说明 + 授权记录”，主线程负责接入和触发策略。
4. 打包与发布分叉只接收主线程确认 ready 的版本，不主动改玩法和素材。
5. 主开发线程每次接入后负责跑通：语法检查、资源路径检查、核心玩法路径、浏览器控制台、关键 UI 交互。
6. 后宫 Tab 角色封面默认锁定，任何分叉不得主动替换，除非主线程明确批准。
7. 每次打包发版由主开发线程和打包与发布分叉互相同步：主线程给出可发版本、缺口和包路径；打包分叉回传发布结果、链接、包体大小和阻塞问题。
8. 分叉发现真实问题时，不要静默修大逻辑；先报告主线程。允许的临时处理只有：补同名占位资源、整理文档、列出缺口、生成待接入素材。
9. 所有分叉产出都必须尽量使用稳定文件名和固定目录，避免主线程二次猜测。
10. 主线程需要定期检查关键分叉健康状态；如果发现分叉读不到、长时间无进展、上下文混乱、连续失败或用户明确说“挂了”，主线程应立即创建同职责新分叉，并把旧分叉归为历史/备用。

## 分叉健康巡检规则

主线程在以下场景必须检查分叉状态：

- 用户要求打包、发布、接入新素材、接入新音频之前。
- 用户说某个分叉“挂了”“没动”“卡住了”“继续不下去”时。
- 美术/剧情、音效/BGM、打包发布任一分叉超过一个工作阶段没有回传时。
- 主线程发现资源缺口、发布阻塞、文件命名不一致、素材产出和剧情计划对不上时。

检查对象：

- 当前剧情和美术分叉：`019ef365-3cd4-7ac1-b4c2-f92abbea470e`
- 音效和 BGM 分叉：`019eef8a-5361-7301-ba7d-b84312549643`
- 打包与发布分叉：`019eefaf-bb7d-7182-aa53-1427433303e1`

判定为“需要重建”的情况：

- 线程无法读取或状态异常。
- 线程长时间停在同一个中间步骤，没有产出或回传。
- 线程已经偏离职责，例如美术分叉开始改发布，打包分叉开始改玩法。
- 线程上下文混乱，无法继续遵守命名、目录、后宫封面锁定等硬规则。
- 连续两次处理同一任务失败，且没有明确可执行的恢复方案。

重建流程：

1. 主线程创建新的同职责分叉。
2. 新分叉标题按职能和序号命名，例如 `女王天下｜美术资产+剧情策划 4号分叉`。
3. 新分叉必须收到：项目路径、主线程职责、当前版本状态、未完成清单、硬规则、交付模板。
4. 主线程更新本文件，把新分叉设为当前主用，把旧分叉移入历史/备用。
5. 主线程向用户报告：旧分叉为何废弃、新分叉 ID、它接下来负责什么。
